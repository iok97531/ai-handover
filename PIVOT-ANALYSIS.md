## 1. 현재 상태 vs 전환 후 상태

### 핵심 정체성 변화

| 항목 | 현재 (개발자용) | 전환 후 (비개발자용) |
|------|-----------------|---------------------|
| **핵심 가치** | "코드베이스를 AI로 이해하고 인수인계한다" | "프로젝트 문서를 AI로 정리하고 지식을 추출한다" |
| **사용자** | 개발자, DevOps, 기술 리드 | PM, 기획자, 마케터, 법무, 경영진 |
| **입력 데이터** | 소스코드 + 설정파일 + README | PDF, Word, PPT, Excel, 이미지, 이메일 |
| **핵심 동작** | 코드 인덱싱 → RAG → 기술 Q&A | 문서 인덱싱 → RAG → 비즈니스 Q&A |
| **UI 언어** | 기술 용어 (인덱싱, 청크, 임베딩) | 비즈니스 용어 (분석, 요약, 검색) |
| **핵심 메타포** | "프로젝트 폴더를 등록하면 AI가 코드를 읽는다" | "문서를 올리면 AI가 내용을 파악한다" |

### 기능별 비교

| 기능 | 현재 | 전환 후 |
|------|------|---------|
| **파일 입력** | 폴더 경로 지정 (로컬 파일시스템) | 드래그앤드롭 업로드 + 폴더 지정 + 클라우드 연동 |
| **지원 포맷** | 40+ 코드 확장자, .md, .txt | PDF, DOCX, PPTX, XLSX, HWP, 이미지(OCR), .md, .txt |
| **파싱** | 텍스트 인코딩 감지, 바이너리 스킵 | 문서 구조 파싱 (제목/본문/표/이미지 텍스트 추출) |
| **청킹** | 800자 고정 분할 | 문서 구조 기반 시맨틱 분할 (섹션, 페이지, 단락) |
| **검색** | 벡터 유사도 검색만 | 벡터 검색 + 키워드 검색 + 메타데이터 필터 |
| **채팅** | 코드 문맥 기반 기술 Q&A | 문서 문맥 기반 비즈니스 Q&A |
| **핸드오버** | 개발자 온보딩, 배포 가이드 등 5종 | 프로젝트 현황 요약, 회의록 정리, 계약 검토 등 |
| **출력** | 채팅 응답 (마크다운+코드 하이라이팅) | 채팅 응답 + 문서 내보내기 (PDF/DOCX) |
| **Git 연동** | 커밋 히스토리 포함 | 불필요 (제거 또는 버전 관리로 대체) |
| **파일 감시** | watchdog으로 코드 변경 실시간 반영 | 문서 폴더 감시 또는 수동 재인덱싱 |

## 2. 기능 변경 상세

### 2.1 문서 파싱 엔진 (가장 큰 기술적 변화)

현재 시스템은 텍스트 파일을 읽고 인코딩을 감지하는 수준이다. 비개발자 문서를 처리하려면 **바이너리 문서 포맷 파싱**이 필수적이다.

#### 필요한 파싱 라이브러리

| 포맷 | 라이브러리 | 특이사항 |
|------|-----------|----------|
| **PDF** | `PyMuPDF`(fitz) 또는 `pdfplumber` | 표 추출, 이미지 내 텍스트는 별도 OCR 필요 |
| **DOCX** | `python-docx` | 표, 스타일, 헤더/푸터 구조 보존 |
| **PPTX** | `python-pptx` | 슬라이드별 텍스트 + 노트 추출 |
| **XLSX/CSV** | `openpyxl` + `pandas` | 시트별 분리, 수식 결과값 추출 |
| **HWP** | `pyhwp` 또는 `olefile` | 한국 시장 필수, 파싱 품질이 불안정할 수 있음 |
| **이미지 OCR** | `pytesseract` + `Pillow` | 스캔 문서, 도면, 캡처 이미지 |
| **이메일** | `email` (stdlib) + `mailparser` | .eml, .msg 파일 지원 |

#### 파싱 아키텍처 변경

```
현재: parser.py → 텍스트 읽기 → 인코딩 감지 → 문자열 반환
변경: parser.py → 포맷 감지 → 전용 파서 호출 → 구조화된 텍스트 반환
                                                    ├─ 제목/섹션 구분
                                                    ├─ 표 → 마크다운 변환
                                                    ├─ 이미지 → OCR 텍스트
                                                    └─ 메타데이터 (작성자, 날짜, 페이지수)
```

### 2.2 청킹 전략 변경

현재 800자 고정 분할은 코드에 최적화된 방식이다. 문서는 **의미 단위로 분할**해야 검색 품질이 올라간다.

| 항목 | 현재 | 변경 후 |
|------|------|---------|
| **단위** | 800자 고정 | 문단/섹션/페이지 기반 가변 |
| **오버랩** | 200자 고정 | 이전 섹션 제목을 컨텍스트로 포함 |
| **메타데이터** | file_path, line_start, line_end | file_path, page_number, section_title, document_type |
| **특수 처리** | 없음 | 표 → 행 단위 분할, 슬라이드 → 슬라이드 단위 |

#### 시맨틱 청킹 예시

```
# 기존: 800자마다 기계적 분할
"...계약 기간은 2024년 1월부터 | 12월까지로 하며, 갱신..."  ← 문장 중간에서 잘림

# 변경: 문단/섹션 경계에서 분할
[청크1] 제3조 (계약 기간) — "계약 기간은 2024년 1월부터 12월까지로 하며..."
[청크2] 제4조 (대금 지급) — "계약 대금은 총 5억원으로..."
```

### 2.3 RAG 파이프라인 고도화

#### 검색 방식 확장

현재는 벡터 유사도 검색만 사용한다. 비개발자는 **정확한 키워드 매칭**(예: 계약번호, 고객명)도 기대한다.

```
현재:  질문 → 임베딩 → ChromaDB 벡터 검색 → top 8
변경:  질문 → ┬─ 임베딩 → ChromaDB 벡터 검색
              ├─ 키워드 추출 → 전문 검색 (BM25 or SQLite FTS)
              └─ 메타데이터 필터 (날짜, 문서유형, 작성자)
              → 결과 병합 + 랭킹 → top 8
```

#### 시스템 프롬프트 변경

현재 시스템 프롬프트는 코드 전문가 페르소나다. 전환 시:

```
현재: "당신은 코드 분석 전문가입니다. 파일 경로와 라인 번호를 인용하세요."
변경: "당신은 프로젝트 문서 분석 전문가입니다.
      출처를 인용할 때는 문서명과 페이지 번호를 명시하세요.
      전문 용어는 쉽게 풀어서 설명하세요.
      표나 수치가 있으면 정리해서 보여주세요."
```

#### 인용 방식 변경

```
현재: [indexer.py:45-67] — 라인 번호 기반
변경: [계약서_2024.pdf, p.12] — 페이지/섹션 기반
      [회의록_0315.docx, 제3항] — 문서 구조 기반
```

### 2.4 UI/UX 전면 재설계

비개발자에게 "인덱싱", "청크", "임베딩" 같은 용어는 이질적이다.

#### 용어 매핑

| 현재 (개발자) | 변경 (비개발자) |
|--------------|----------------|
| 프로젝트 인덱싱 | 문서 분석 |
| 인덱싱 중... 45% | 문서를 읽고 있어요... 45% |
| 2,340 chunks indexed | 2,340개 구간 분석 완료 |
| File watcher active | 새 문서 자동 감지 중 |
| Embedding model | AI 분석 엔진 |
| API Key 설정 | AI 연결 설정 |
| RAG 검색 | 문서에서 답변 찾기 |

#### 핸드오버 포맷 → 문서 분석 템플릿

현재 5종 핸드오버 포맷을 업종/용도별로 재구성:

**범용 템플릿:**
| 템플릿 | 설명 |
|--------|------|
| 프로젝트 현황 요약 | 전체 문서를 종합해서 프로젝트 상태를 정리 |
| 핵심 문서 요약 | 주요 문서의 핵심 내용만 추출 |
| Q&A 보고서 | 특정 질문에 대한 근거 기반 답변 |
| 비교 분석 | 여러 문서의 내용을 비교 정리 |

**업종 특화 템플릿:**
| 업종 | 템플릿 예시 |
|------|------------|
| **법무** | 계약서 주요 조항 요약, 리스크 조항 검토, 계약 비교 |
| **마케팅** | 캠페인 성과 종합, 경쟁사 분석 정리, 고객 피드백 분류 |
| **경영/기획** | 사업계획서 요약, KPI 달성 현황, 이해관계자 보고서 |
| **HR** | 채용 현황 종합, 교육 이수 현황, 인사 정책 Q&A |
| **연구/학술** | 논문 요약, 선행 연구 비교, 실험 데이터 정리 |

#### 새로운 UI 요소

1. **문서 뷰어** — 원본 문서를 앱 내에서 미리보기 (PDF 렌더링, DOCX 변환 표시)
2. **드래그앤드롭 업로드** — 폴더 경로 지정 외에 파일 직접 드래그 지원
3. **문서 목록/태그** — 등록된 문서를 유형별, 날짜별, 태그별로 정렬/필터
4. **내보내기 버튼** — AI 답변을 PDF/DOCX로 내보내기
5. **하이라이트** — AI가 참조한 원본 문서 구간을 시각적으로 표시

### 2.5 문서 내보내기 기능 (신규)

비개발자에게 "채팅 기록"보다 **문서화된 결과물**이 더 중요하다.

```
AI 분석 결과 → 마크다운 → PDF 변환 (WeasyPrint 또는 pdfkit)
                       → DOCX 변환 (python-docx)
                       → 공유 링크 (Phase 2)
```

템플릿 기반 출력:
- 보고서 형태 (표지, 목차, 본문, 출처)
- 요약 형태 (1~2페이지 브리핑)
- 비교표 형태 (항목별 대조)

### 2.6 보안 및 접근 제어 강화

비개발자 환경에서는 **문서 보안**이 코드 보안보다 더 엄격할 수 있다.

| 항목 | 현재 | 필요 |
|------|------|------|
| 사용자 인증 | 없음 (로컬 앱) | 로그인, 비밀번호, SSO |
| 문서 접근 권한 | 없음 | 프로젝트별 열람/편집 권한 |
| 감사 로그 | 없음 | 누가 언제 어떤 문서에 질문했는지 기록 |
| 데이터 보존 정책 | 없음 | 자동 삭제, 보존 기간 설정 |
| 워터마크 | 없음 | 내보내기 시 워터마크 옵션 |

## 3. 비즈니스 모델 변경

### 3.1 타겟 시장 확장

#### 현재 타겟 (TAM이 좁음)

```
개발자 → 코드 인수인계가 필요한 팀
         ├─ 프리랜서 (프로젝트 마감 시)
         ├─ 스타트업 (직원 퇴사/합류 시)
         └─ SI 업체 (프로젝트 이관 시)
```

#### 전환 후 타겟 (TAM이 급격히 확대)

```
비개발자 → 문서가 많은 모든 조직
           ├─ 법무팀 — 계약서, 법률 자문, 규정
           ├─ 마케팅팀 — 캠페인 자료, 보고서, 리서치
           ├─ 경영기획 — 사업계획서, IR 자료, 이사회 자료
           ├─ HR — 인사규정, 교육자료, 채용 문서
           ├─ 연구소 — 논문, 실험 보고서, 특허
           ├─ 컨설팅 — 고객 산출물, 제안서, 분석 보고서
           ├─ 교육기관 — 교재, 연구자료, 행정 문서
           └─ 공공기관 — 정책 문서, 감사 자료, 민원 기록
```

### 3.2 경쟁 환경 변화

#### 현재 경쟁 (개발자 도구)

직접 경쟁자가 적고, 코드 인수인계는 틈새시장이다.

| 경쟁자 | 차별점 |
|--------|--------|
| GitHub Copilot | 코드 작성 도구지 인수인계 도구가 아님 |
| Cursor | 에디터 기반, 인수인계 관점 없음 |
| Swimm | 코드 문서화 도구, 가장 유사하지만 인수인계 특화 아님 |

#### 전환 후 경쟁 (문서 AI)

경쟁이 **매우 치열**해진다. 이미 큰 투자를 받은 플레이어들이 존재한다.

| 경쟁자 | 규모 | 차별점 |
|--------|------|--------|
| **Notion AI** | 시리즈C, $10B 밸류 | 문서+AI 통합, 이미 거대한 사용자 기반 |
| **ChatPDF / ChatDOC** | 수백만 사용자 | PDF 특화 AI Q&A, 무료 진입 |
| **Glean** | $4.6B 밸류 | 기업 내부 검색 AI, 대기업 타겟 |
| **Guru** | 인수됨 | 사내 지식 관리 + AI |
| **Mem.ai** | 시리즈A | 개인 지식 관리 AI |
| **Microsoft Copilot** | 무한 자원 | M365 통합, 기업 시장 장악력 |

#### 경쟁에서 살아남으려면

단순히 "문서에 AI Q&A를 붙이는" 것으로는 차별화가 불가능하다. 가능한 차별화 축:

1. **완전 오프라인/온프레미스** — 기존 Enterprise 전략의 연장선. 규제 산업에서 클라우드 기반 경쟁자가 못 들어가는 영역
2. **한국 시장 특화** — HWP 지원, 한국어 최적화, 국내 규정 준수, 한국 기업 문화에 맞는 템플릿
3. **특정 버티컬 집중** — 법무, 컨설팅 등 특정 업종에 깊게 특화 (범용 도구와 싸우지 않는다)
4. **데스크톱 네이티브** — 브라우저 기반 경쟁자 대비 로컬 파일 접근이 자연스러움

### 3.3 가격 체계 재설계

#### 현재 가격 (개발자용)

```
Free: 프로젝트 1개, 본인 API 키
Pro: $19/월, 무제한 프로젝트
Enterprise: 협의 (오프라인, 사내 LLM)
```

#### 전환 후 가격 (비개발자용)

비개발자는 "API 키"라는 개념 자체를 모른다. **API 키 직접 입력 모델을 유지할 수 없다.**

#### 핵심 변화: AI 비용 구조

| 항목 | 현재 | 전환 후 |
|------|------|---------|
| AI 비용 부담 | 사용자 (본인 API 키) | 서비스 (구독료에 포함) |
| 마진 구조 | AI 비용 0 | AI 비용이 COGS의 대부분 |
| 수익성 | 높음 (비용이 거의 없음) | 사용량 관리가 핵심 |
| 진입장벽 | 높음 (API 키 발급 필요) | 낮음 (가입 즉시 사용) |

**AI 비용 관리 전략:**

1. **사용량 기반 과금** — 기본 쿼터 + 초과분 별도 과금
2. **모델 티어링** — Free는 저렴한 모델(Haiku), Pro는 고성능 모델(Sonnet)
3. **캐싱** — 동일/유사 질문 결과 캐싱으로 중복 호출 절감
4. **청크 최적화** — 불필요한 컨텍스트를 줄여 토큰 사용량 절감
5. **API 키 직접 입력 옵션** — 고급 사용자용으로 유지 (비용 절감 옵션)

## 6. 단계별 전환 로드맵

### 권장: 점진적 확장 (하이브리드 전략)

완전 전환보다 **기존 개발자 기능을 유지하면서 문서 관리 기능을 추가**하는 것을 권장한다. 이유:

1. 개발자 시장에서의 선점 우위를 버릴 필요가 없다
2. 기술 문서(README, 설계서)는 개발자와 비개발자 모두 사용한다
3. Enterprise 타겟은 동일하다 (오프라인, 보안)
4. 점진적 확장은 리스크가 낮다

### Phase A — 문서 포맷 확장 (4~6주)

> 기존 개발자 기능은 유지하면서, 비코드 문서도 인덱싱할 수 있게 한다

**백엔드:**
- [ ] PDF 파서 추가 (`PyMuPDF`)
- [ ] DOCX 파서 추가 (`python-docx`)
- [ ] PPTX 파서 추가 (`python-pptx`)
- [ ] XLSX/CSV 파서 추가 (`openpyxl` + `pandas`)
- [ ] `config.py` 지원 확장자 업데이트
- [ ] `MAX_FILE_SIZE` 50MB로 확장
- [ ] 문서 메타데이터 추출 (작성자, 날짜, 페이지수)

**프론트엔드:**
- [ ] 문서 유형별 아이콘 표시
- [ ] 인덱싱 진행률에 문서 유형 정보 추가

**검증:** 기존 코드 프로젝트 + PDF/DOCX가 혼합된 폴더에서 정상 인덱싱 및 Q&A 확인

### Phase B — 문서 중심 UX 추가 (4~6주)

**백엔드:**
- [ ] 시맨틱 청킹 옵션 추가 (문서 유형 감지 → 자동 전략 선택)

**검증:** 타겟 업종 실제 기업 2~3곳에서 파일럿 테스트

- [ ] 사용자 인증 (로그인/회원가입)
- [ ] 프로젝트별 접근 권한
- [ ] 감사 로그
- [ ] 팀 공유 기능
- [ ] HWP 파싱 (한국 시장 공략)
- [ ] OCR (스캔 문서 지원)
- [ ] Enterprise 라이선스에 문서 관리 기능 포함

## 7. 결론 및 권장 전략

### 완전 전환은 비추천

- 개발자 코드 인수인계 시장에서의 **틈새 선점 우위를 포기**하게 된다
- **레드오션** (Notion AI, ChatPDF, Glean, Microsoft Copilot)에 진입하는 것은 소규모 팀에게 불리하다
- AI 비용 구조가 **수익성을 악화**시킨다

### 하이브리드 확장을 추천

**"코드 + 문서를 모두 이해하는 유일한 프로젝트 AI"** 라는 포지셔닝이 가장 강력하다.

이유:
1. **차별화**: ChatPDF는 코드를 모르고, GitHub Copilot은 문서를 모른다. 둘 다 하는 도구가 없다
2. **자연스러운 확장**: 개발 프로젝트에는 코드와 문서가 공존한다 (설계서, 기획서, API 문서, 회의록)
3. **Enterprise 강점 유지**: "코드도 문서도 외부로 나가지 않는다"는 더 강력한 세일즈 포인트
4. **기존 자산 활용**: 파서, 청커, RAG, 벡터DB 등 핵심 인프라를 그대로 재사용
5. **점진적 검증**: 시장 반응을 보면서 방향을 조절할 수 있다

### 최종 권장 순서

```
[현재] 개발자 코드 인수인계 도구
   ↓ Phase A (문서 포맷 확장)
[+4주] 코드 + PDF/DOCX/PPTX를 모두 인덱싱하는 프로젝트 AI
   ↓ Phase B (UX 이중화)
[+8주] 개발자/비개발자 모두 사용할 수 있는 프로젝트 AI
   ↓ Phase C (검색 + 내보내기)
[+12주] 실무에서 쓸 수 있는 문서 분석 도구
   ↓ Phase D (업종 특화)
[+18주] 특정 업종에서 "이 도구 없이는 안 된다"는 평가
   ↓ Phase E (Enterprise)
[+26주] 보안이 중요한 조직의 프로젝트 지식 관리 플랫폼
```

이 전략의 핵심은 **기존 것을 버리지 않으면서 새로운 것을 얹는 것**이다. 각 Phase마다 시장 반응을 확인하고, 문서 관리 쪽 수요가 확인되면 비중을 늘리고, 반응이 미미하면 개발자 도구에 집중하면 된다.
